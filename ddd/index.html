DDD 是什么?

初识 DDD

"DDD" = Domain Driven Design

"领域驱动设计"

一套指导软件设计与实施的方法论

关键术语

术语意义说明
领域 问题空间 某一类业务问题, 比如"如何发布内容"
限界上下文 解空间 对应于某一个领域的解决方案, 比如内容发布系统
领域模型 对领域内的对象建模, 封装业务逻辑 内容发布系统中的"评论"对象, 具有删除动作
实体对象 领域模型的一种, 具有全局唯一性 "发布系统"上下文中的"评论"对象是实体对象
值对象 领域模型的一种, 生命周期短, 不唯一 "发布系统"上下文中的"评论"对象的发布者 ID 是值对象

举例说明

对于"社区内容"问题(领域):

我们定义出"帖子上下文"(限界上下文);

在该上下文中, 我们有帖子这一(实体)对象;

该对象可能包含的属性有"帖子 ID", "帖子作者", "帖子标题", "帖子内容"

这些属性都是值对象

产研实践的缺陷

  • 业务定义不清晰
  • 业务用语有歧义
  • 业务逻辑丢失
  • 业务扩展困难

DDD 的优点

  • 以业务为核心, 围绕业务开展活动

  • 建立全员一致的业务理解

  • 更好的团队投入与参与(产品, 运营, 开发者, etc)

  • 更好的发掘/梳理业务价值, 聚焦高价值的业务

  • 代码即业务逻辑

  • 代码更加易于维护,扩展;(可能)更高的代码质量

如何实施 DDD?

For everybody

  1. 梳理业务逻辑
  2. 输出共识

1. 梳理业务逻辑

这是 ddd 开始阶段要做的第一件事

我们可以使用名为"EVENT STORMING"的技术来帮助我们梳理业务

EVENT STORMING 步骤参考

  1. 确定要讨论的业务
  2. 召集相关方聚在有白板的会议室中
  3. 通过brainstorm 提出业务的事件, 贴上白板
  4. 如果事件 A 触发了事件 B, 连接它们
  5. 重复3-4步

最后我们就得到了业务的全部流转逻辑

1. event storming 是对业务逻辑的梳理, 对于"创建合集"操作来说, 就是该操作完整的业务流程以及各种逻辑分支

2. event storming 需要我们集合领域专家采纳听取他们的意见, 厘清业务逻辑中的要点

2. 输出共识

共识即领域通用语言

通用语言可能包含:

  • 领域 和 限界上下文
  • 用语词典
  • 业务流程图

领域 & 限界上下文

领域= 问题空间

限界上下文= 解空间 需要由团队人为划定边界

例如, 对于帖子合集功能

"领域" = 帖子合集的所有功能点

"限界上下文" = "合集上下文", 包含

  1. 对合集操作的实际工作流程(交互与代码实现)
  2. 对帖子与合集关系的操作的实际工作流程

2. 为领域建模

原则

  • 创建纯洁的领域模型
  • 将 IO 推到系统边界
纯洁的领域对象应该 1. 对应于领域中的一个术语 2. 单一职责(SRP) 3. 不知道任何自己职责以外的知识

"值对象"

  • 不唯一
  • 生命周期短
  • 不可变
1. 请试着说出这段代码的业务逻辑 2. "值对象"的概念介绍: 在业务中不具备唯一性的对象, 比如通常意义上的姓名, 用户所在城市, 一段用户输入的文案等; 特点包括 1. 生命周期短 - 需要结合实体对象来说明, 暂时认为, 只有值对象是无法存在或者具有意义的 2. 不可变

"实体对象"

  • 唯一性/身份
  • 生命周期长
  • 可变
						0. 首先简要介绍一下 kotlin 语法
							1. val = final
							2. var = mutable prop
							3. ? = @Nullable
						1. 请观众说明该段代码的业务含义
							1. 成员对象的可变性(val, var)
							2. 可变属性的变更=替换操作 , 当我们修改某个属性时, 直接将该属性对象替换为一个新的值对象即可

							3. Compilation 对象是可变的, 体现在其可变成员属性上
